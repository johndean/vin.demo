# VIN AI Demo — Failure Root-Cause Investigation

> Forensic, evidence-grounded audit (2026-06-13). Principal-architect panel (AI · Product · Voice · Frontend · UX · Systems · Observability + Enterprise Demo). 6 parallel forensic readers over the real code → synthesis. Method: drill why→why→why to the ARCHITECTURAL cause; cite `file:line`; no solutions until root cause is established. Distinct from `DEMO_EXPERIENCE_AUDIT.md` (felt-quality) and `DEMO_ROOT_CAUSE_AUDIT.md` (architecture/works-and-safe): this one explains the *live* "select journey → speaks one paragraph → stops → becomes chat" failure.

## Reported failure (the symptom to explain — not assumed to be the cause)

Select Journey → **long delay** → AI speaks → **first paragraph only** → speech **stops** → transition **breaks** → **text appears** → user forced to **read** → conversation becomes a **chat** → flow/momentum/magic **die**.

## Executive summary

The reported failure (select journey → long delay → AI speaks one paragraph → speech stops → text appears → it becomes a chat) is not a chain of independent bugs. It is the predictable, deterministic output of ONE architectural decision: VIN was built as a stateless, single-LangGraph-loop question-and-answer engine with a chat-panel UI, and the "journey walk" — the thing meant to be a guided AI-consultant experience — was bolted on TOP of that Q&A engine as a deterministic narrated screen-tour that reuses the SAME loop, the SAME chat surface, and the SAME operator-paced defaults. Nothing in the system knows it is in "consultant mode." Verified in source: the right Convo panel opens by default and the Conversation tab is active by default (runtime.tsx:1428-1429); the message reducer blindly appends EVERY spoken line as a chat bubble with zero spoken-vs-text discrimination (runtime.tsx:860, rendered 755-761); auto-advance defaults OFF and the only toggle is buried in the operator Control Room, never the buyer Talk runtime (runtime.tsx:1411, 1488, 1686), so after step 1 the walk waits for an operator click that the buyer cannot see or send (voice-session.ts:267 fires step 0, then 301-302 require journey_next); first audio is gated behind a serial boot→interpret→embed→pgvector→navigate→narrate chain (live-session.ts:103-138, graph.ts:263-284, llm.ts:399-429 interpret has NO thinking control so adaptive thinking blocks the first word); and there is NO facilitator engine at all — discover() is explicitly SKIPPED on every walk turn (graph.ts:325) and the dramatic arc is a one-time caption-assembly computed by POSITION (journey-assembler.ts:120, journeys.ts:153-168), not an executable state machine. The "magic dies" because the surface IS a chat interface, the brain IS a one-shot Q&A loop, and the journey IS a static playlist — none of the three was rebuilt for a live, continuous, buyer-facing consultative experience. Recent Wave A→D work (cold-start bridge, teleprompter, auto-advance dwell timing, Gemini streaming, ASK→TALK memory) is almost entirely cosmetic or gated OFF in the live buyer path: it polished the tour and masked the silence without touching the three root systems. The honest verdict: this is a missing-layer problem, not a tuning problem. The CIO should fund Option A (a real AI-Consultant runtime) and explicitly retire the chat-panel-first, one-loop, static-playlist foundation underneath the journey.

## Top 10 root causes (ranked)

### 1. [CRITICAL] The journey walk reuses the Q&A chat surface verbatim — text is a first-class, always-visible competing anchor
*Category:* UX / Surface Architecture  
*Evidence:* runtime.tsx:1428 const [panelOpen,setPanelOpen]=useState(true); :1429 const [tab,setTab]=useState('convo'); :860 reducer blindly appends every {type:'message'} as a bubble; :755-761 Convo renders all messages; control-room.css:86 fixed 372px panel vs :67 stage flex:1. No useEffect closes the panel when isJourneyWalk (runtime.tsx:1486 defines isJourneyWalk but nothing acts on it for the panel).  
*Why it matters:* This is THE reason it 'feels like a chatbot.' Every spoken line is simultaneously printed as a persistent chat bubble in an always-open right panel, while the product stage is passive. The operator's/buyer's eye is pulled to the moving text stream. Speech becomes a voice-over of a transcript instead of the primary channel. No mode awareness means the same layout that's correct for ASK/TALK Q&A is catastrophically wrong for a guided walk.

### 2. [CRITICAL] No facilitator engine — the 'consultant' is a deterministic narrated screen-tour, not a consultative dialogue
*Category:* Demo Flow / Facilitation  
*Evidence:* graph.ts:314 navigateJourneyStep runs only on journeyAdvance; :325 'discover: skipped on a journey walk step (presenter-led narration, no discovery interrupt)'; journey-assembler.ts:120 'arcRole ... computed downstream by POSITION — so the ORDER here IS the arc'; journeys.ts:153-168 arcRole assigned by index; live-session.ts:323 committee objections matched ONLY on off-script grounded objection turns, never in the walk.  
*Why it matters:* The spec demands Consultant/Facilitator/Discovery Expert/Outcome Advisor; the architecture delivers 'Guided Product Tour with Voice.' Buyer pain, objections, and signals are logged but DECOUPLED — they never reshape the next screen. There is no discovery turn, no committee dialogue, no re-routing, no 'what do you think?' beat. The narration is canned because it WAS assembled once and is played back unchanged. This is the missing layer the whole product is supposed to be.

### 3. [CRITICAL] Auto-advance defaults OFF and is invisible to the buyer — so the walk stalls in silence after step 1
*Category:* Voice / Walk Driver  
*Evidence:* runtime.tsx:1411 useState(false); :1488 if(!isJourneyWalk||!autoWalk) return; (dwell timer never arms); :1686 Auto-advance checkbox only rendered inside the isJourneyWalk operator block, not the Talk runtime; voice-session.ts:267 fires step 0 via void runWalkStep(); :301-302 every subsequent step requires {type:'journey_next'} from the client.  
*Why it matters:* This is the literal mechanism of 'speech STOPS after paragraph 1.' Step 0 auto-runs, then the engine waits forever for an operator click the buyer can neither see nor send. The shipped Wave #34 dwell/teleprompter fixes are fully implemented but DEAD CODE because they're gated behind autoWalk=true, which is never set in the buyer path. Correct code, wrong (and unreachable) configuration.

### 4. [CRITICAL] One stateless LangGraph loop conflates reasoning with orchestration — each LLM call's streaming window is local and dies between phases
*Category:* AI Orchestration  
*Evidence:* graph.ts:263-264, 282-284 narrate() completes inside the node and returns explanation; live-session.ts:231 single graph.invoke per turn; :291-293 explanation emitted as a 'message' AFTER invoke returns (streaming window already closed); voice-session.ts:138-156 say_chunk TTS-only, :162 message spoken only if !streamedThisTurn — a partial stream that then errors leaves streamedThisTurn=true and the fallback is SUPPRESSED, so the buyer hears half a sentence then silence.  
*Why it matters:* There is no 'speak continuously while thinking' driver. narrate and answerAs are independent calls with independent streaming windows; state (recentNarrations, committee, activeStakeholder) collapses between them. On any LLM timeout/partial stream there is no recovery — the boolean streamedThisTurn flag can't represent an incomplete stream. This is why speech fragments and why off-script answers arrive as sudden text avalanches.

### 5. [CRITICAL] First-audio is gated behind a fully serial boot→interpret→embed→retrieve→navigate→narrate chain (8-15s)
*Category:* Performance / Latency  
*Evidence:* live-session.ts:103-138 serial boot DB cascade (product→journey→outcome→committee) all awaited before turn 1; graph.ts StateGraph whoSpeaks→interpret→retrieve→navigate serialized; llm.ts:399-429 interpret has NO thinking config (defaults adaptive → thinking block gates the first word); retrieval.ts:145-148 embed then pgvector serial; graph.ts:282 Promise.all only covers driveTo vs narrate on narrated node steps.  
*Why it matters:* This is the 'LONG DELAY' in step 1 of the report. The architecture never treated first-audio as a critical metric. interpret's adaptive thinking alone adds 1-2s of pure latency tax for a classification the user never sees. Nothing is pre-staged on journey select; every step re-runs the full serial chain.

### 6. [HIGH] The cold-start bridge masks the silence but creates a two-burst stutter (bridge → silence → real narration)
*Category:* Performance / Voice  
*Evidence:* voice-session.ts:256-261 speakColdStartBridge emits a templated line; :267 queued on ttsChain BEFORE void runWalkStep(); the bridge end and the real narrate() ready-time are independent events with no alignment — runWalkStep then awaits the full interpret→retrieve→navigate→narrate chain.  
*Why it matters:* This is a verified instance of recent work treating the symptom, not the cause. The buyer hears 'Let me bring up [product]…', then a SECOND silence while the real first narration is composed, then narration — perceived as a stutter/failure, which is precisely the 'transition breaks' in the report. It is social engineering over the latency, not a latency fix.

### 7. [HIGH] First walk step's explanation can render as text WITHOUT audio (or audio without the matching bubble suppressed)
*Category:* Voice / Rendering Coordination  
*Evidence:* live-session.ts:230 onDelta→say_chunk set when turn.stream; :291-293 out.explanation always emitted as a message event regardless; voice-session.ts:135-156 say_chunk TTS-only and NOT forwarded; :162-164 message spoken only if !streamedThisTurn but ALWAYS forwarded to the client. Gemini (non-streaming) emits no say_chunk (llm.ts:161 path) → full blocking narration with no chunking.  
*Why it matters:* This is steps 6-7 of the report: 'TEXT appears → user forced to READ.' The forwarding of the message to the chat panel is NOT gated on whether it was already spoken, so text appears in the Convo regardless. With Gemini selected the narration doesn't stream at all, so the latency is fully exposed.

### 8. [HIGH] Narration state does not span the walk — silent transits wipe recentNarrations; the answer path never sees what was narrated
*Category:* AI Orchestration / State  
*Evidence:* graph.ts:252 recent=state.recentNarrations; :302 recentNarrations:[say] for a narrated beat but [] for a silent/recovery beat; live-session.ts:345-346 answerAs receives audience+priorContext but NOT recentNarrations; recentNarrations reset per-turn (not on the checkpoint boundary).  
*Why it matters:* Over a 10-step journey the narration becomes a series of isolated observations, never a story — it can't say 'as I showed you' or 'building on that.' A silent transit between two narrated beats erases continuity. Off-script answers don't know the walk already covered something, producing repetition and incoherence that reinforce the 'canned' feel.

### 9. [HIGH] Committee objections and ROI are SCORING SIGNALS baked into static captions, not live dialogue participants
*Category:* Demo Flow / Facilitation  
*Evidence:* journey-assembler.ts:130-155 painMember/painObjection/painCriterion extracted ONCE and composed into captions; graph.ts:263,284 narrate paraphrases the caption; live-session.ts:322-332 committee concern matching exists but only on off-script grounded objection turns (k==='objection' && hasSource), which the walk never reaches (journeyAdvance forces discover skip).  
*Why it matters:* The intelligence the product invested in (role-based committee, decision criteria, quantified outcomes) is MUTE during the walk — it shaped which chunk was picked at assembly time but never speaks as a live concern. A real CFO objection in the room does not pivot the walk. This is the 'intelligence built but mute' tell from the prior experience audit, now traced to the walk path specifically.

### 10. [MEDIUM] Every off-script question pays a 3-6s navigation + answer penalty with no parallelization or prefetch
*Category:* Performance / Latency  
*Evidence:* graph.ts:147-172 driveTo serial open()→gotoNode()→screenshot() on the Playwright path (clientNav defaults false, live-session.ts:100); navigate blocks answerAs on free-roam turns (no Promise.all outside narrated node steps); retrieval.ts:69-83 4-table JOIN + pgvector LIMIT 4 every turn; embeddings never cached/batched (retrieval.ts:145-148).  
*Why it matters:* When a buyer DOES barge in (the only way to make the walk feel interactive today), they hit 3-6s dead air for navigation then 2-4s for the answer, confirming 'slow, robotic' perception. Combined with the busy guard (voice-session.ts:170-171), interrupting the walk feels broken — which pushes the operator back toward the safe, dead, scripted playback.

## Systemic root cause — the single architectural mistake

VIN was built as a stateless, single-LangGraph-loop question-and-answer engine with a chat-panel UI, and the journey walk — the experience that is supposed to BE the AI consultant — was bolted on top of that Q&A engine rather than built as its own runtime. The journey reuses the same one-shot loop (live-session.ts:231 single graph.invoke per turn), the same chat surface (runtime.tsx:1428-1429 panel-open + convo-tab default, 860 undiscriminating message reducer), and the same Q&A defaults (autoWalk OFF, operator-paced, runtime.tsx:1411/1488/1686). Because the walk is just 'the Q&A engine fed a pre-assembled playlist,' it inherits all three failure modes at once: (1) it RENDERS as chat because the surface is a chat panel and every spoken line is appended as a bubble; (2) it STALLS because the loop is turn-based and operator-paced with auto-advance off and invisible to the buyer; (3) it has NO FACILITATION because the loop is stateless per turn and discover() is skipped on walk turns (graph.ts:325), so the arc is a static caption playlist (journey-assembler.ts:120, journeys.ts:153-168) not a live state machine. The single mistake is the absence of a dedicated AI-Consultant RUNTIME — a continuous-speech, mode-aware, stateful facilitation layer — that OWNS the experience instead of borrowing the Q&A engine. Every symptom in the report (delay → speak → stop → text → chat → momentum dies) is the deterministic projection of that one missing layer. The evidence that this is the keystone: each fix shipped in Wave A→D addressed ONE downstream symptom (bridge for silence, teleprompter for operator preview, dwell for advance, streaming for first-sentence audio) and NONE moved the felt experience, because they all live downstream of a foundation that is structurally a chatbot.

## Honest verdict on recent (Wave A→D + follow-on) work

Brutally honest: roughly one of five recent investments helped the felt experience, and even that one was neutralized downstream. HELPED (partially): Gemini/say_chunk sentence-streaming on the answer path (live-session.ts:358, llm.ts:596-604) is the single best architectural decision — it correctly keeps say_chunk TTS-only and starts speech on sentence 1 with thinking disabled. But it's neutralized because (a) the full message still arrives and is rendered as a chat bubble (runtime.tsx:860), and (b) Gemini itself doesn't stream (llm.ts:161 path), so under the default-switchable model the benefit can vanish. MISSED THE FELT EXPERIENCE: (1) Cold-start bridge #15 (voice-session.ts:256-267) masks silence but creates a two-burst stutter and does not touch the serial boot/narrate latency — it's social engineering, not a fix. (2) Teleprompter + auto-advance dwell #34 (runtime.tsx:1488-1509) is fully correct code that is DEAD in the buyer path because autoWalk defaults false and the toggle is operator-only (runtime.tsx:1411/1686) — it solved dwell-timing for an operator clicking Next while the buyer has no operator and no button. (3) Unified walk driver #19 gave eval/voice parity (excellent for correctness) but both paths are deterministic tours, so unification added zero facilitation. (4) ASK→TALK memory (driveHistory/driveNarrative) solved cross-modality continuity, orthogonal to the chatbot-feel and the walk stall. (5) Wave C committee/ROI/framedFor enrichment (journey-assembler.ts:130-155, live-session.ts:348-354) enriched WHAT is said but never WHEN/HOW, and the committee intelligence stays mute in the walk (live-session.ts:323 fires only off-script). The pattern is unmistakable and matches the prior memory audits: the team kept polishing the tour and masking symptoms while the three root systems (chat surface, one-loop brain, static-playlist journey) went untouched. Several of these fixes also indict each other — the dwell timer and teleprompter assume an operator that the live buyer flow does not have. The work was diligent but aimed at the surface, not the foundation.

## CIO verdict: **A-AI-Consultant**

The reported failure is not a polish gap — it is the absence of the product's core layer. Options C (Advanced Product Demo) and D (Chatbot with voice) are exactly what the system ALREADY is and what is failing: C is the deterministic narrated tour the walk already implements (graph.ts:314, journeys.ts:153-168), and D is the chat-panel-first surface that makes it 'feel like a chatbot' (runtime.tsx:1428-1429,860). Doubling down on either would re-fund the foundation that produces the failure. Option B (Guided Product Experience) is closer but is still framed as guidance layered on the tour; it would likely reproduce the operator-paced, static-playlist behavior with nicer captions — i.e., more Wave-A→D-style polish. Only Option A (AI Consultant) names the missing layer: a stateful, mode-aware facilitation RUNTIME with a continuous-speech driver and live discovery that re-shapes the walk. The evidence that A is required and not aspirational: discover() is skipped on walk turns (graph.ts:325), the arc is static (journey-assembler.ts:120), committee intelligence is mute in the walk (live-session.ts:323), and the surface renders speech as chat (runtime.tsx:860) — every one of these is a consultant-runtime responsibility that no current component owns. Fund A; explicitly sunset the chat-panel-first surface, the one-loop-for-everything brain, and the static-playlist journey as the demo path (keep them as the operator/eval substrate only).

## Architectural root causes (ranked)

- SINGLE LOOP CONFLATES REASONING WITH ORCHESTRATION: one graph.invoke per turn (live-session.ts:231) runs interpret→retrieve→navigate→discover→END serially; every LLM call (narrate, answerAs) completes inside its node and its streaming window closes before the next runs (graph.ts:263-284, llm.ts:601/639). There is no conversation-level speech driver that can speak continuously across phases.
- THE JOURNEY IS A STATIC PLAYLIST, NOT A RUNTIME: journeyWalkPlan expands story_flow into a flat WalkEntry list (journeys.ts:134-168) with arcRoles assigned by POSITION; journey-assembler.ts:120 confirms 'the ORDER here IS the arc.' The dramatic arc (pain→stakes→show→proof→close) is an authoring discipline frozen at assembly time, never an executable facilitation state machine.
- NO MODE BOUNDARY BETWEEN Q&A AND CONSULTANT: the walk reuses the Q&A engine, the Q&A chat surface (runtime.tsx:1428-1429,860), and the Q&A defaults (autoWalk OFF, panel open). Nothing in the system branches on 'I am now a guided consultant' — isJourneyWalk exists (runtime.tsx:1486) but only toggles minor UI, not the layout hierarchy or the loop's behavior.
- DISCOVERY IS A PASSIVE LOG, NOT AN ACTIVE ROUTER: discover() is explicitly skipped on every walk turn (graph.ts:325) and recordDiscovery writes to the DB with no state update or re-route; buyer signals never feed back into screen/proof selection.
- FIRST-AUDIO WAS NEVER A DESIGN CONSTRAINT: serial boot cascade (live-session.ts:103-138), serial walk-plan DB loop (journeys.ts:134-147), interpret with adaptive thinking and no streaming (llm.ts:399-429), and no prefetch/cache of the first step's narration, embedding, or node resolution.

## Performance root causes (ranked)

- Serial boot DB cascade before turn 1: product→journey→outcome→committee all awaited (live-session.ts:103-138) before any audio prep.
- journeyWalkPlan is a sequential for-loop with one DB round-trip per workflow/knowledge step (journeys.ts:134-147) instead of a parallel Promise.all.
- interpret runs with default adaptive thinking and no streaming (llm.ts:399-429); the thinking block gates the first spoken word with 1-2s of latency the user never sees value from.
- narrate is the true critical path to first audio (graph.ts:263-284); Promise.all only parallelizes driveTo vs narrate on narrated node steps, and driveTo is fast while narrate's LLM round-trip is the bottleneck.
- No prefetch/cache: the first step's narration prompt, knowledge chunk, node selectors, and embedding are not pre-staged during the cold-start bridge (voice-session.ts:212-267).

## If VIN was rebuilt correctly (the requested target-state deliverable)

A world-class AI-Consultant experience from journey-select to demo-complete, rebuilt as a dedicated runtime (NOT the Q&A loop + chat panel):

TIMING. Journey-select must reach FIRST WORD in under ~1.2s and feel like a person clearing their throat, not a system booting. Achieve it by (a) pre-staging on select: while a genuinely warm, NON-templated greeting streams ('Thanks for making time — let me pull up [product] and walk you through how [outcome] actually lands'), the runtime fires boot DB queries in parallel (product/outcome/committee independent — fix live-session.ts:103-138), parallelizes journeyWalkPlan's per-step queries (Promise.all, fix journeys.ts:134-147), and prefetches step 0's narration prompt + knowledge chunk + node selectors + embedding; (b) disabling adaptive thinking on interpret and streaming it (llm.ts:399-429); (c) decoupling the first narration from navigation so a real warm open streams immediately while step 0's grounded narration composes in the background. The bridge stops being a band-aid and becomes the genuine first beat, time-aligned to hand off seamlessly to the composed narration (no second silence).

VOICE. ONE continuous speech driver owns the whole session, not per-node streaming windows that collapse. Speech never stops waiting for an operator: the consultant finishes a beat and NATURALLY continues or NATURALLY hands the floor ('…that's the approval bottleneck. Does that match what you're seeing?') — auto-advance is the default behavior of the runtime, not a checkbox (delete the autoWalk=false default, runtime.tsx:1411). streamedThisTurn becomes a completion marker so a partial/failed stream is gracefully completed or repaired, never left as half a sentence into silence. A single TTS chain spans phases so narration→answer→narration is one monologue.

NAV. The product is the stage and the star; the consultant drives it as it talks. driveTo is client-nav by default (no 3-6s server Playwright in the buyer path), pre-resolved during prefetch, and time-aligned so the screen change lands ON the sentence that references it ('watch what happens when I approve this…' → click).

AI / FACILITATION. A real facilitator state machine: OPEN (frame the committee + outcome) → DISCOVER (ask, listen, and ROUTE the answer — discover() runs DURING the walk and its signals re-rank the next proof/screen; reverse graph.ts:325) → SHOW (grounded demonstration) → OBJECTION (when the live CFO speaks, the committee intelligence that today is mute (live-session.ts:323) answers AS that role, with the matched objection + decision criterion + ROI number, then resumes) → PROOF → CLOSE (next steps). The arc is EXECUTABLE and stateful, not a position-indexed caption playlist (replace journeys.ts:153-168 + journey-assembler.ts:120). State (what was narrated, what the buyer raised, which committee concerns are still open) is threaded through every call (fix the recentNarrations reset, graph.ts:302; pass it to answerAs, live-session.ts:345).

JOURNEY. Still assembled zero-gap from real assets (keep the discipline) — but assembly produces a FACILITATION SCRIPT with branch points and dialogue beats, not a flat WalkEntry list. The buyer's answers change the path.

UX. Mode-aware surface. On journey start the layout INVERTS: the product stage goes full-bleed and primary; the chat panel CLOSES (the missing useEffect on isJourneyWalk, runtime.tsx:1486). Spoken lines are NOT rendered as chat bubbles by default (gate the reducer, runtime.tsx:860) — at most an ephemeral, low-weight live-caption under the stage that fades, so text supports speech instead of competing with it. A transcript is available on demand (a tab the buyer can open), never the default anchor. The result: the buyer WATCHES the product and LISTENS to a consultant who sees what they see, adapts to what they say, and never makes them read — the chat panel, the one-shot loop, and the static playlist are gone from the demo path entirely.
